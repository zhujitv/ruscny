# 账号注销与数据删除

## 1. 用户入口

正式账号必须在 App 内提供可发现的自助入口：

```text
设置 → 账号与安全 → 注销账号
```

入口不能只写“联系客服”或只跳转邮件。页面先说明影响，再要求用户主动确认。访客可在同一入口删除当前会议范围内的临时身份；该操作不会全局禁用同一设备在其他会议的合法访客身份。

## 2. 注销前说明

确认页至少显示：

- 所有设备将立即退出，当前会议连接中断。
- 注销后不能再用原账号访问历史会议。
- 正式账号只有统一的注册用户类型。注销会立即结束该用户作为 owner 创建且仍在进行的会议；其只作为参会者加入的他人会议不会被结束。
- 账号和参会资料会匿名化，解除与 Participant/Contact 的直接账号关联，并以稳定匿名标识保留“谁说了什么”的序列与引用关系；已有发言文本、顺序及其他有权限者依法保留的业务记录继续只读可用。
- 删除进入备份传播和第三方清理流程，可能存在明确的最长处理周期。
- 法律要求保留的数据会隔离、限制用途并在期限届满后删除。
- 注销不可仅通过卸载 App 完成；卸载不会自动删除服务端账号。

任何具体处理天数、撤销窗口和依法保留期限必须在上线前由运营主体确认，并与隐私政策、后端任务和客服口径一致。

## 3. 身份验证

为防止冒名删除：

1. 要求当前会话最近完成过认证；否则重新输入密码或验证码。
2. 显示账号的脱敏手机号/邮箱，说明删除对象。
3. 要求二次确认，例如输入“注销账号”或勾选不可恢复说明。
4. 高风险异常设备、刚修改安全信息或 Token 状态异常时，可要求额外验证或人工审核。

不得要求用户提供与账号无关的身份证明。账号被禁用或无法登录时，仍需提供联系渠道和合理的身份核验流程。

## 4. HTTP 合同

当前第一版 API：

```http
DELETE /v1/auth/account
Authorization: Bearer <access-token>
Content-Type: application/json
```

正式账号在最近 10 分钟内完成密码登录/注册时，body 可为空。超出该窗口时返回 `401 RECENT_AUTH_REQUIRED`，App 应重新收集密码并提交：

```json
{ "password": "current-account-password" }
```

密码错误返回 `401 INVALID_CREDENTIALS`。Guest 没有密码，使用当前会议范围内、未失效的 Bearer 凭证确认要删除的临时身份。

成功：

```json
{
  "ok": true,
  "data": {}
}
```

当前实现会锁定相关会议并在 PostgreSQL 事务内完成身份撤销与匿名化：

- `User.status` 转为 `DELETED`，邮箱、手机、密码哈希、公司和头像清空，所有设备 Refresh Token 撤销。
- Participant 与 User/GuestIdentity 的直接外键解除，姓名和公司改为匿名快照；TranslationMessage 保留 `participantId`、原文、译文、语言、时间和 `sequence`，仅匿名化发言人的可识别字段。已持久化的会议纪要中参会名单、发言快照和待办负责人也使用同一匿名标识。
- 参会者和访客邮箱快照被清空；历史纪要邮件收件人快照改为匿名标识，尚未发送或发送中的任务转为失败，避免注销后继续发信。
- 注册用户注销时将其作为 `ownerId` 创建的 `WAITING/ACTIVE` 会议终结，对其他有权参会者保留只读历史。
- Guest 只撤销当前 scoped GuestIdentity，不会把其 GuestPrincipal 在其他会议中全局封禁。
- 成功提交后 API 广播该用户所拥有会议的结束事件，并断开注销主体的 Socket。

账号注销路径不删除作为共享会议记录一部分的 TTS 引用，避免因某一参会者注销破坏其他参会者的完整历史。系统现在会写入可查询的 `DataDeletionRequest` 在线删除台账，记录数据库匿名化、设备撤销和共享记录处理状态；Redis、第三方和备份传播仍需继续补充执行器与独立完成证据，不能因为台账显示在线步骤完成就宣称全副本删除完成。

安全要求：

- 只能删除当前认证主体，API 不接受客户端提供 `userId`。
- 重复请求不得泄露旧账号数据；账号已经删除后原 Token 会失效，因此客户端不能假定还能用同一 Token取得第二次 200。
- 正式账号使用 `DELETED + deletedAt` 软删除墓碑，不再依赖 User 外键级联删除会议数据。
- 删除提交后立即断开该用户的 Socket.IO 连接。
- 响应后 App 清除 Access/Refresh Token、本地 SQLite 缓存、访客身份、待上传音频和推送 Token。
- 失败响应不得暴露对象 key、bucket、其他用户或内部存储细节。

App 只有收到成功响应并清除本地数据后才能显示在线账号已注销。失败时 App 必须保留登录状态；对 `RECENT_AUTH_REQUIRED` 应提示密码重认证，不得循环重试空 body。PostgreSQL、Redis、第三方和备份仍不是一个原子系统，传播尚无完成证据。

## 5. 完整删除增强目标

当前的事务化匿名路径不等同面向账号全生命周期的完整 deletion request。以下流程仍是正式上线应补齐的目标。

### 立即执行

1. 写入审计事件和 `deletionRequestId`，不在日志写敏感正文。
2. 将 User 状态设为不可登录/删除处理中。
3. 撤销并删除所有设备 Refresh Token、Push Token 和活跃会话。
4. 关闭其正在参与的实时连接；结束或冻结其作为 owner 创建的活动会议。
5. 阻止新建客户、会议、邀请、翻译和导出。

### 异步清理

依外键顺序清理或匿名化：

- 用户个人资料、认证凭据、验证码/密码重置数据和 UserDevice。
- 用户拥有的 Contact、Conversation、Participant、TranslationMessage、Glossary、未来 Summary、导出和审计中的可删除个人字段。
- 对应对象存储中的 TTS 音频及其资产引用；原始上传音频当前只在单次请求内存缓冲中处理，不产生独立长期文件。
- Redis presence、房间、限流、幂等和缓存 key。
- APNs/FCM、监控或其他受托服务中的可定位数据。

注册用户注销而某会议属于另一名创建者时，不能简单级联删除该会议的完整业务记录。上线前应确定合法基础和产品规则：优先删除注销账号资料、解除 `linkedUserId`，并对 Participant 显示名/个人标识做匿名化；若用户依法有权删除会议内容，则进入人工或规则化删除流程。该决定必须写入正式隐私政策。

### 完成

1. 对每个数据存储记录成功/失败和重试次数。
2. 所有必需步骤完成后把任务标为 `COMPLETED`。
3. 仅保留最小化、不可用于恢复账号的删除审计证明。
4. 如保留联系渠道且有合法依据，可发送完成通知；通知中不包含会议内容。

## 6. 备份和恢复

在线数据库和对象存储完成删除后，备份中的副本在既定生命周期内隔离并自然过期，不为日常业务恢复单个已删除账号。若灾难恢复使用旧备份，系统必须在开放流量前重新执行 deletion ledger，避免注销账号和已删会议复活。

账号注销墓碑和单次会议的持久资产清理任务会随数据库备份恢复，但从早于删除请求的旧备份恢复时仍可能缺失。完整跨存储 deletion request、已完成任务保留期和备份保留周期必须纳入恢复演练。

## 7. 会议创建者删除单次会议

账号注销之外，创建会议的注册用户应能在会议详情执行“删除会议”：

1. 明确显示客户、主题、日期和消息数，二次确认。
2. 服务端验证当前用户是 `ownerId`，不信任 App 传入角色。
3. 删除/匿名化该 `conversationId` 下 Participant、Message、Summary 占位、导出和 TTS 对象。
4. Redis 关闭房间并广播 `room.ended`/资源不可用。
5. 同一删除请求幂等，已删除资源返回稳定结果，不泄露其是否属于其他用户。

非 owner 注册参会者和 Guest 无权删除整个会议，但可按隐私政策请求处理其个人信息。

## 8. App Store / Play 验收

- 从登录账号不超过三次点击可到达注销入口。
- 注销前身份校验、影响说明和二次确认完整。
- 提交后所有设备 Refresh Token 失效，旧 Access Token 最迟在短期过期/撤销检查后失效。
- 当前 Socket 断开，本地安全存储和 SQLite 清空，返回未登录页。
- 再次使用原凭据不能直接恢复已注销账号。
- 上线增强后的删除请求覆盖 PostgreSQL、Redis、对象存储和第三方推送/监控，并提供可验证状态。
- owner 与非 owner 注册参会者的数据归属已经法律审核并有自动化测试。
- 备份恢复后 deletion ledger 能重新应用。

## 9. 客服处理模板所需信息

用户无法登录时，客服只收集最少信息：脱敏账号线索、注册方式、最近使用时间、设备和请求内容。禁止通过普通邮件索取密码、验证码、Access/Refresh Token、API Key 或完整会议内容。人工操作需双人审批或等价控制、记录审计并告知预计处理时间。
